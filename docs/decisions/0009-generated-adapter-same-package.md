# 0009. Generate the Go adapter into the handler's own package

**Status:** Accepted
**Date:** 2026-05-17

## Context

The `--go-adapter` generator emits a `Register(chi.Router)` plus per-route
`http.HandlerFunc` wrappers that must call the user's handlers and construct
their request structs. If the generated file lived in a separate package, it
could only reach exported symbols, forcing handlers and request types to be
exported and risking import awkwardness. The chi-basic golden output settles
this: `expected/go/goduct_routes.go` is `package api` — the same package as
`api/users.go` — header-marked "Code generated by goduct. DO NOT EDIT.", and
its `Register` wires unexported `handleXxx` wrappers that call the handlers
directly. The README's `main.go` calls `api.Register(r)`.

## Decision

The go-adapter generator writes `goduct_routes.go` into the **same package**
as the handlers it wraps, so `Register()` and its wrappers reference handlers
and request structs directly without exporting them or importing across
packages.

## Consequences

- Easy: no import cycle, no exported-symbol requirement; the adapter is a
  single drop-in file per handler package.
- Hard / giving up: the generated file shares the package's build and test
  scope — a generation bug breaks the whole package build (relevant to the
  `go build ./...` concern raised earlier in this conversation). The generated
  file occupies the package namespace, so wrapper names (`handleXxx`) can
  collide with hand-written code.

## Alternatives considered

- Separate generated package importing the handler package — rejected: forces
  all handlers and request types to be exported; awkward for unexported
  handlers.
- Runtime reflection/registration — rejected: loses static wiring and the
  compile-time guarantee that every route is connected.
