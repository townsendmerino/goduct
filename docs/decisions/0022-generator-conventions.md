# 0022. Generator conventions

**Status:** Accepted
**Date:** 2026-05-18
**Amended:** 2026-05-18 — §2: Go header is one-line (matches the frozen
golden); source string derived from `api` via `internal/gen.SourcePath`,
not a CLI side channel

## Context

Per [0003](0003-generators-as-pipeline.md), each generator is an
independent package consuming `ir.API`. The analyzer is now complete and
the generator milestones are next (tstypes, zod, tsclient, goadapter).
Several decisions cut across all generators and should be pinned once
rather than re-decided per generator.

## Decision

All generators adhere to the following conventions.

### 1. Public API shape

Every generator package exports exactly one function:

```go
func Generate(api *ir.API, w io.Writer) error
```

The generator writes its output to `w`. Multi-file generators (none in
v0.1) would expose `Generate` variants per file, but each generator in v0.1
emits a single file's worth of content.

Generators do not return strings, do not take filenames, do not handle the
filesystem. The CLI (separate milestone) is responsible for opening files
and wiring writers.

### 2. File header

Generated TypeScript files begin with a two-line header:

```
// Code generated by goduct. DO NOT EDIT.
// source: <package-import-path>
```

Generated Go files begin with the standard go-generated marker on a
single line, followed by a blank line, then the package clause (gofmt's
required placement):

```
// Code generated by goduct. DO NOT EDIT.

package <name>
```

Rationale for the divergence: Go's tooling has a long-established
`"Code generated ... DO NOT EDIT."` regex (see `go/build`'s
`IsGenerated`) that triggers off the single-line form. Adding a second
comment line doesn't break detection, but the canonical Go form is one
line and matches what `go generate`-style tools in the ecosystem produce.
TypeScript has no equivalent convention; the `source:` line is useful
provenance and costs nothing.

The package-import-path value is derived from the `api` (every route and
type in the IR carries it via qualified names), not passed in. Generators
derive it via `internal/gen.SourcePath` (see §6); the CLI does not provide
it. This keeps the generator signature (§1) uniform and avoids a CLI-time
side channel.

### 3. Determinism

Generator output must be deterministic across runs for the same input.
Specifically:

- Routes are emitted in `ir.API.Routes` order (source order).
- Types are emitted in a stable order derived from a topological sort of
  the type dependency graph, with ties broken by qualified name.
  Generators implement this sort once in a shared helper (see §6).
- Map iteration is never used to drive output order. Map lookups for
  individual values are fine; iteration order is not.

### 4. Formatting

Generators emit code that is already correctly formatted for its target
language. Specifically:

- **Go generators:** output must pass `gofmt` unchanged. Use
  `go/format.Source` on the final byte slice as the last step inside
  `Generate`. A `gofmt` failure on generator output is a generator bug,
  not a user problem; return the format error wrapped with context.
- **TS generators:** output must be Prettier-compatible (2-space indent,
  double quotes, trailing commas in multi-line, semicolons present). We do
  not invoke Prettier at generation time (avoiding the dep); the
  generator's templates produce already-Prettier-clean output by
  construction. The golden tests are the enforcement mechanism.

### 5. Error handling

Generators return errors via the standard error return. They do not log,
they do not panic on user-caused conditions. They may panic on
internal-invariant violations (e.g. a `TypeRef` pointing at a name not in
`api.Types` — that's an analyzer bug manifesting in the generator, and
should be loud).

User-facing errors at the generator layer are rare in v0.1 because the IR
is already validated by the analyzer. Most generator errors will be I/O
errors from `w.Write`.

### 6. Shared helpers

A new internal package, `internal/gen/`, holds helpers used by multiple
generators:

- Topological sort of types by dependency.
- Filtering a `TypeDef`'s fields to "wire-visible" fields
  (`Source == FieldSourceJSON` only).
- Grouping routes by `Tag` for client emit.
- `SourcePath(api)` — the package import path for the `// source:` header
  line (§2); comma-joined + sorted for future multi-package input.
- Translating `ir.TypeRef` to target-language type strings is **NOT**
  shared — each generator handles its own target language's idioms.

`internal/gen/` is created by the first generator milestone (tstypes);
subsequent generators add to it.

### 7. Testing

Each generator has both unit tests (small synthetic `ir.API` values
testing specific rendering rules) and golden tests (run `Generate` against
the chi-basic `ir.API`; diff bytes against `testdata/expected/...`). Golden
tests are the v0.1 acceptance criterion; unit tests are for fast iteration
during development.

The golden test for each generator lives in the generator's own package;
the cross-cutting end-to-end golden test (running the full pipeline) lives
in a separate test package and is the last milestone before v0.1 release.

### 8. No cross-generator dependencies

Generators do not import each other. If a piece of logic looks shared
between two generators, it goes into `internal/gen/`. This preserves the
parallel-generation model from [0003](0003-generators-as-pipeline.md).

## Consequences

- The CLI's contract with generators is uniform and trivial: pick a
  writer, call `Generate`.
- Adding a new generator (svelte, swift, kotlin) is mechanical: new
  package, implement `Generate`, add to the CLI's generator list.
- The determinism rule means we can golden-test confidently without flaky
  map-ordering failures.
- Forcing `gofmt` at generation time means we never ship a Go generator
  that produces `gofmt`-dirty output, which would be embarrassing for a Go
  tooling project.

## Alternatives considered

- Generators take a config struct — rejected; v0.1 needs no
  configurability and adding it preemptively invites scope.
- Generators write files directly — rejected; couples generators to
  filesystem layout and complicates testing.
- Skip the `gofmt` step, rely on the user to format — rejected; the whole
  point of code generation is that the output is correct, not that the
  user has to clean it up.
